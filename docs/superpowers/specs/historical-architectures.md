# OmniScraper Pro 历史演进方案存档

本文档记录了 OmniScraper Pro 在产品定义阶段探讨过的不同架构方案与功能设想，作为未来二次开发或架构升级的参考。

## 方案A：中心化 SaaS 架构（服务器 + Web前端）

**适用场景**：希望实现真正的“多端同步查看”、“账号配额管理”、“管理员统一控制”的商业化 SaaS 产品。

### 1. 架构设计
*   **爬虫 Worker 集群**：Python + Playwright 部署在一台或多台云服务器上（或使用动态代理池）。
*   **中心后端 API**：处理用户登录、任务分发、配额扣减、数据存储（PostgreSQL）。
*   **Web 前端面板**：响应式 Web 应用（React/Vue），用户在手机/电脑浏览器访问即可操作。

### 2. 核心特性
*   **账号体系与配额**：
    *   完善的注册/登录机制（邮箱、微信扫码等）。
    *   普通账户（每日免费 N 次查询）、特种战士账户（付费订阅，无限额或高限额）。
*   **多端同步与跨设备**：
    *   云端存储所有抓取结果和分析报告。
    *   用户在电脑端提交抓取任务，任务在云端后台运行，完成后可在手机端直接查看生成的 Markdown 报告和知识库。
*   **管理员控制台**：
    *   全局查看所有用户的任务状态、资源消耗。
    *   封禁/解封用户，手动调整用户配额等级（提升为特种战士）。
    *   查看全局风控报警（如某个节点的 IP 被抖音封禁）。

### 3. 为什么最终被放弃？
*   **开发与运维成本高**：需要维护云服务器、数据库、代理池。
*   **合规与法律风险**：在云端集中存储大量抓取的用户数据（尤其是评论数据）面临更高的数据隐私合规要求。
*   **用户隐私担忧**：用户可能不愿意将抓取的数据（或自己配置的 LLM API Key）上传到中心服务器。

## 方案B：分布式微服务架构（适合大规模抓取）

**适用场景**：企业级数据采集，需要极高吞吐量和极强防风控能力。

### 1. 架构设计
*   **调度中心**：负责任务拆分（例如将一个关键词的 1000 个视频分配给 10 个节点同时抓取）。
*   **边缘节点**：可以部署在树莓派、闲置 PC、廉价 VPS 上，每个节点负责执行具体的浏览器自动化任务。
*   **消息队列**：如 RabbitMQ/Kafka，连接调度中心与边缘节点。

### 2. 核心特性
*   **极致防风控**：利用分布式节点的天然 IP 隔离，降低单一出口被封禁的概率。
*   **高吞吐量**：并发抓取多个平台的视频和评论。

### 3. 为什么最终被放弃？
*   **过度设计**：对于个人工具或小团队使用来说，架构过于复杂，部署难度极大。

---

## 方案C：混合架构（本地运行 + 云端同步）

**适用场景**：既想享受本地运行的隐私/零服务器成本，又想要一定程度的跨端查看能力。

### 1. 架构设计
*   **本地桌面端**：承担 100% 的爬虫执行、LLM 调用和数据存储（SQLite）。
*   **云端同步服务（可选）**：支持绑定 WebDAV、阿里云 OSS、或私有化部署的简单同步服务器。

### 2. 核心特性
*   **数据所有权归用户**：抓取和分析全在本地。
*   **多端查看妥协方案**：用户手动点击“同步到云端”，然后在手机上通过一个配套的轻量级 H5 页面（连接到同一个云存储）读取 Markdown 报告和 SQLite 数据。

### 3. 为什么最终被放弃？
*   **用户体验割裂**：需要用户自己配置云存储（门槛高），或者开发者提供云存储（产生服务器成本，违背了纯本地初衷）。最终决定退回到最纯粹的方案一（仅通过导出文件分享）。

## 附加功能灵感池（待未来实现）

*   **智能去重与增量更新**：再次搜索相同关键词时，对比本地 SQLite，只抓取“上次任务之后新发布的视频”或“新增加的评论”，大幅节省 LLM Token 和抓取时间。
*   **人工标注闭环**：在本地报告视图中增加“点赞/点踩”按钮。如果 LLM 判定某条评论是干货但用户觉得不是，可以点击“误判”，收集这些数据用于未来微调本地分类模型或优化 Prompt。
*   **知识库自动标签归类**：利用 LLM 将零散的知识点自动聚类成体系化的思维导图（例如把所有关于“剪辑技巧”的知识点融合成一篇长文）。
*   **多模态增强（VLM）**：预留了接口。未来如果本地多模态模型（如 LLaVA）能在普通显卡上流畅运行，可直接让模型“看”视频提取视觉信息（如 PPT 内容、操作演示步骤），而不仅仅依赖 OCR。
